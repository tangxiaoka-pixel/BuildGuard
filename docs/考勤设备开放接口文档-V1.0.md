# BuildGuard 考勤设备开放接口文档 V1.0

本文档用于考勤设备厂家把签到、心跳、通行考勤数据双发到 BuildGuard 智慧工地平台。

## 接口地址

推荐地址：

- `POST /open/cwr/Signin`
- `POST /open/cwr/KeepAlive`
- `POST /open/cwr/UploadPassedLog`

兼容人社平台路径：

- `POST /CWRService/Signin`
- `POST /CWRService/KeepAlive`
- `POST /CWRService/UploadPassedLog`

第一期预留但暂不开放：

- `QueryEmpListHash`
- `QueryEmployeeList`
- `QueryEmployeeInfo`
- `QueryEmployeeIdInfo`

上述预留接口返回 `code=501`。

## 公共参数

公共参数放在 query string：

| 参数 | 必填 | 说明 |
| --- | --- | --- |
| `api_key` | 是 | 接入账号 |
| `api_version` | 是 | 接口版本，建议传 `1.0` |
| `client_serial` | 是 | 设备序列号，用于匹配设备 |
| `timestamp` | 是 | `yyyy-MM-dd HH:mm:ss`，允许 10 分钟误差 |
| `signature` | 是 | MD5 签名 |

Body 使用 JSON。

## 签名规则

参与签名字段：

- `api_key`
- `api_version`
- `body`
- `client_serial`
- `timestamp`

字段按名称升序拼接 `字段名 + 字段值`，首尾拼接 `api_secret`，再做 MD5，结果转大写。

示例：

```text
api_secret + api_key{api_key} + api_version{api_version} + body{原始JSON字符串} + client_serial{client_serial} + timestamp{timestamp} + api_secret
```

注意：`body` 使用原始 JSON 字符串，不做字段排序，不做 URL 编码。

默认联调账号：

```text
api_key=buildguard-demo
api_secret=buildguard-secret
```

## 统一响应

```json
{
  "result": "true",
  "status": 200,
  "code": "00",
  "message": "",
  "detail_message": "",
  "result_data": {}
}
```

常见错误码：

| code | 说明 |
| --- | --- |
| `00` | 成功 |
| `24` | 缺少签名参数 |
| `25` | 签名错误 |
| `29` | 无效或停用的 `api_key` |
| `30` | 缺少 `timestamp` |
| `31` | `timestamp` 超出允许范围或格式错误 |
| `32` | 缺少 `api_version` |
| `40` | 必填业务参数缺失 |
| `41` | 参数类型或时间格式错误 |
| `43` | 参数或业务处理错误 |
| `47` | JSON 报文解析失败 |
| `501` | 接口已预留，第一期暂不开放 |

## Signin 设备签到

设备启动或重新上线时调用。

请求 body 可为空或 `{}`。

成功响应会返回服务器时间、是否需要人员同步、设备绑定项目信息。

未知 `client_serial` 会自动创建待确认设备，数据不会丢失。

## KeepAlive 设备心跳

设备定时调用，用于更新在线状态和最后心跳时间。

请求 body 可为空或 `{}`。

## UploadPassedLog 通行考勤上传

支持两种格式。

单条格式：

```json
{
  "emp_id": "E10001",
  "emp_name": "张三",
  "id_code": "330681199001011234",
  "emp_company": "杭州强盛劳务有限公司",
  "team_name": "钢筋班组",
  "work_typename": "钢筋工",
  "pass_time": "2026-06-22 08:12:30",
  "pass_direction": "in",
  "verify_type": "face",
  "score": 92
}
```

兼容人社文档批量格式：

```json
{
  "passedlog_list": [
    {
      "up_from": "device",
      "emp_company": "杭州强盛劳务有限公司",
      "person_type": "worker",
      "person_id": "E10001",
      "person_name": "张三",
      "passed_time": "2026-06-22 08:12:30",
      "direction": "in",
      "way": "face",
      "site_photo": ""
    }
  ]
}
```

方向字段：

- `in` / `IN` / `进场`：进场
- `out` / `OUT` / `出场`：出场

## 数据归属规则

1. 系统按 `client_serial` 查找设备。
2. 设备不存在时，自动创建“待确认设备”，仍保存接口日志和原始考勤。
3. 设备绑定项目和区域后，后续考勤自动归属到该项目和区域。
4. 设备未绑定项目时，考勤仍保存，`match_status=DEVICE_UNBOUND`。
5. 人员匹配顺序：
   - 身份证号 `id_code`
   - 外部人员 ID `emp_id/person_id` + 项目
   - 姓名 + 单位 + 工种
6. 未匹配但有姓名和身份证号时，系统自动创建待确认劳务工人，`match_status=AUTO_CREATED`。
7. 无法匹配且缺少身份证号时，只保存原始考勤，`match_status=UNMATCHED`。

所有开放接口请求都会写入 `device_open_api_log`，原始考勤会写入考勤记录的 `raw_data`。
